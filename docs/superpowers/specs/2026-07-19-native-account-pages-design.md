# 原生账户页面设计

## 背景

Vibe Usage 当前侧栏只有用量、排行榜、远程日志、同步和设置。用户需要在同一个 macOS 应用内管理 new 系统令牌与钱包，并预留活动中心入口。新增页面必须沿用应用现有的 SwiftUI 视觉语言，不嵌入或复刻 new 系统网页布局。

## 目标

- 在侧栏新增“账户”分组，依次包含“令牌管理”“钱包管理”“活动中心”。
- 令牌管理通过 new 系统现有 `/api/token` 接口完成用户自己的令牌查询和管理。
- 钱包管理通过 new 系统现有用户与充值接口展示余额、消耗、请求次数、充值配置和充值记录，并把支付交给外部浏览器完成。
- 活动中心在后端功能缺失期间提供稳定的原生空状态，后续可原位接入真实活动数据。
- 所有页面只在用户进入、手动刷新或完成变更后读取数据，不增加后台定时轮询。

## 非目标

- 不复制 new 系统 React/Semi UI 页面或其布局。
- 不使用 WebView 承载令牌、钱包或活动页面。
- 不伪造活动、活动倒计时、奖励或领取结果。
- 不在客户端长期保存完整令牌密钥、支付参数或充值订单敏感信息。
- 不新增 new 系统数据库扫描接口。

## 信息架构与导航

侧栏保持现有“数据”和“应用”分组，在二者之间增加“账户”分组：

1. 令牌管理，图标使用 `key.horizontal`。
2. 钱包管理，图标使用 `wallet.pass`。
3. 活动中心，图标使用 `gift`。

`DashboardPage` 增加对应枚举值、标题和副标题。三个入口均使用现有整行可点击的 `sidebarItem`，选中态与现有页面一致。页面切换时关闭用量筛选浮层。账户页面不参与用量或排行榜自动刷新协调器，远程刷新目标保持 `.none`。

## 共享数据与网络边界

### API 客户端

在现有 `APIClient` 上增加类型安全方法，继续复用 URLSession 的 HttpOnly 登录 Cookie 与 `New-Api-User` 请求头：

- `GET /api/token/?p={page}&size={size}`：分页读取令牌。
- `GET /api/token/search`：按名称或令牌搜索。
- `POST /api/token/`：创建令牌。
- `PUT /api/token/`：编辑令牌；`status_only=true` 时只切换状态。
- `DELETE /api/token/{id}`：删除令牌。
- `POST /api/token/{id}/key`：仅在用户主动点击时读取完整密钥。
- `GET /api/user/self`：刷新余额、历史消耗和请求次数。
- `GET /api/user/topup/info`：读取支付开关、最低充值额和可选金额。
- `GET /api/user/topup/self?p={page}&page_size={size}`：分页读取充值记录。
- 支付创建继续使用 new 系统当前启用的支付接口。获得支付 URL 或表单参数后交给外部浏览器，不在应用内渲染 new 系统页面。

所有响应映射为独立 Swift 模型。服务端返回业务失败时显示服务端 `message`；401 继续触发现有远程会话清理逻辑；429 保留服务端冷却时间，不自动重试。

### 页面状态

令牌与钱包分别使用聚焦的 `@MainActor @Observable` 页面模型，负责加载状态、错误、分页和变更中的按钮禁用。视图只负责展示和发送用户动作，避免继续扩大 `DashboardShellView` 与 `AppState`。AppState 只提供当前已认证客户端能力和未授权处理入口。

页面模型遵循：

- 首次进入加载一次。
- 返回页面时保留已加载数据，不重复请求。
- 用户点击刷新时立即请求一次。
- 创建、编辑、启停、删除或完成支付跳转后，仅刷新受影响页面。
- 同一加载或变更操作不能并发重复提交。

## 令牌管理页面

### 布局

页面使用现有 20pt 内容边距、7pt 圆角、`AppTheme.surface` 卡片和细边框：

1. 顶部三张摘要卡：令牌总数、已启用数量、有限额度合计。
2. 工具栏：名称/令牌搜索框、“刷新”和“新建令牌”。
3. 原生表格：名称、脱敏密钥、状态、已用/剩余额度、模型限制、到期时间、操作。
4. 底部分页：上一页、页码、下一页及总数。

宽窗口展示完整表格；窄窗口保持最小表格宽度并允许水平滚动，不压缩文字到不可读。

### 操作

- 新建和编辑使用应用风格的 Sheet，不复制网页弹窗。
- 表单支持名称、无限额度/有限额度、剩余额度、到期时间、模型限制、IP 白名单、分组和跨分组重试。
- 状态切换直接调用 `status_only=true`，失败时恢复原状态并显示错误。
- 删除前显示确认对话框。
- 列表始终只显示服务端脱敏密钥。
- “复制密钥”必须由用户主动点击；成功读取完整密钥后直接写入剪贴板，不写日志、不进入持久化状态，提示完成后立即释放局部值。

## 钱包管理页面

### 布局

1. 顶部三张摘要卡：当前余额、历史消耗、请求次数。
2. 充值卡：后端允许的预设金额、手动金额、支付方式与预计支付金额；未启用在线充值时显示明确空状态。
3. 充值记录表：订单号、支付方式、充值额度、支付金额、状态、创建时间。
4. 记录分页与手动刷新。

颜色、字体、间距、表格行高和状态标记均复用当前应用主题。支付成功、待支付、失败和过期只通过轻量状态文字与小圆点区分，不采用网页彩色大卡片。

### 支付边界

- 应用只负责校验输入、请求后端创建订单并启动外部浏览器。
- Stripe、Creem、Waffo 等直接 URL 使用 `NSWorkspace.open`。
- 需要表单 POST 的支付方式由专门的 `ExternalPaymentLauncher` 构建一次性提交页并交给默认浏览器；临时内容使用用户私有权限，启动后删除，不记录参数。
- 应用不判断支付最终结果；用户返回后通过“刷新”读取充值记录和账户余额。

## 活动中心页面

活动中心使用完整正式页面而不是空白：

- 页面标题和副标题由顶部栏提供。
- 主内容是一张应用风格空状态卡，包含 `gift` 图标、“暂无活动”和“新活动上线后会在这里显示”。
- 不显示虚假活动卡片、结束时间、奖励金额或领取按钮。
- 不发起任何网络请求。

后续后端上线后，活动模型、加载状态和活动卡可作为独立模块加入，不需要修改导航结构。

## 错误、安全与负载

- 页面加载失败保留已有数据，并在顶部显示可重试错误条。
- 删除和支付创建属于不可重复操作，按钮在请求期间禁用。
- 完整令牌密钥不进入 `AppState`、磁盘缓存、调试日志或错误信息。
- API 调试日志只记录方法与路径，不记录请求体和支付返回参数。
- 页面没有 Timer、后台 scheduler 或自动翻页；因此不会持续增加服务器和数据库负载。
- 搜索由用户提交触发，不按每个键盘输入请求。

## 测试与验收

### 自动测试

- 导航测试覆盖三个新页面的标题、副标题、整行点击和页面映射。
- APIClient 测试覆盖令牌分页、创建、编辑、启停、删除、主动读取密钥、钱包摘要、充值配置与记录分页。
- 模型测试覆盖状态、额度、到期时间和充值状态格式化。
- 页面状态测试覆盖首次加载去重、手动刷新、变更后刷新、失败保留旧数据和防重复提交。
- 源码契约测试确认活动中心不发起请求、列表不保存完整密钥、账户页不加入自动刷新协调器。

### 界面验收

- 侧栏新增三个入口，整行均可点击，选中态与现有页面一致。
- 明暗模式下卡片、表格、弹窗和空状态具有足够对比度。
- 令牌在未主动读取时始终脱敏，复制后剪贴板内容正确。
- 钱包金额、余额和充值记录与 new 系统返回一致。
- 活动中心只显示真实空状态。
- 拉宽、缩窄和滚动窗口时没有明显卡顿，也不会产生持续网络请求。

## 交付方式

实现按导航骨架、账户 API 模型、令牌页、钱包页、活动页、集成验证的顺序进行。每一阶段先写失败测试，再写最小实现。最终运行全量测试、Release 构建、安装到 `/Applications/Vibe Usage.app` 并逐页进行界面验证，然后提交并推送当前分支。
