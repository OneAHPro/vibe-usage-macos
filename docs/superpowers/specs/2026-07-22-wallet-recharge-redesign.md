# 钱包充值界面重设计

## 目标

把余额充值从暴露内部核算步骤的表单，改成符合桌面支付习惯的一键流程：用户选择支付方式和充值金额后，点击一次主按钮即可看到软件内支付二维码。同时让“当前订阅”和“余额充值”在双栏布局中严格等高，提升钱包页的整体完成度。

## 现状问题

- “预计支付金额 / 请先计算 / 计算实付”把服务端金额核算暴露成额外操作，用户无法直接充值。
- 充值表单由多个小控件和并列按钮拼接，主次关系弱，像后台调试工具。
- 当前订阅为空时自然高度较短；自定义双栏布局虽用较高卡片计算行高，卡片内容本身没有填满分配高度。
- 支付方式、金额、快捷金额与主操作缺乏连续的阅读和操作顺序。

## 方案比较

### 方案 A：保留金额预览，改为输入后自动请求

能继续展示折扣后的金额，但每次输入或切换金额都可能触发接口请求，增加服务器负载，也容易出现防抖、竞态和闪烁。

### 方案 B：一次点击完成核算并创建订单（采用）

卡片只展示支付方式、充值金额和“立即充值”。点击后在同一加载状态中依次完成服务端金额核算与订单创建，二维码弹窗显示最终实付金额。请求次数与旧流程相同，但不再把核算作为用户步骤。

### 方案 C：跳过核算直接创建订单

交互最短，但无法可靠显示折扣或渠道调整后的实付金额，也可能违背现有支付接口契约。

## 交互设计

余额充值卡片按以下层级组织：

1. 标题和说明：明确“下一步直接显示支付二维码”。
2. 支付方式：保留原生选择控件，但提升宽度和点击区域；Creem 继续显示产品选择。
3. 充值金额：金额输入成为卡片主视觉，带货币符号；快捷金额紧邻输入区域，当前值有明确选中态。
4. 主操作：使用整宽强调按钮“立即充值”；处理中显示“正在创建支付订单…”。
5. 安全说明：明确二维码在软件内展示，不跳转浏览器，也不自动轮询。

移除“预计支付金额”“请先计算”“计算实付”和双按钮区域。

## 数据流

- 用户编辑金额或支付方式时不发送请求。
- 点击“立即充值”后：
  - Creem 使用产品已知价格；
  - 其他渠道先调用现有金额核算接口，得到最终实付金额；
  - 随后调用现有下单接口；
  - 成功后把支付地址、支付方式和最终实付金额传给二维码弹窗。
- 任一步失败均保留现有错误横幅，不打开空弹窗。
- 全流程共用一个 `isMutating` 状态，防止重复下单。
- 不新增轮询、自动刷新或输入时请求。

## 布局设计

- 新增统一的钱包概览卡片最小高度。
- “当前订阅”和“余额充值”根容器接受自定义布局分配的行高，并以顶部对齐方式填满。
- 默认窗口继续双栏；最小窗口继续单列，单列时不强制两张卡片相互等高。
- 卡片内部使用更稳定的垂直节奏，主按钮固定在内容底部，空订阅状态占满可用空间。

## 视觉方向

沿用应用现有原生浅色/深色主题与绿色费用强调色，不引入新的品牌色。重点风险放在金额输入区：使用更大的等宽数字、克制的绿色选中态和整宽主按钮，形成“支付工作台”而不是拼装表单。其余区域保持安静，避免装饰性图标和渐变。

## 测试

- 商店层：一次准备支付会先核算金额、再创建一次订单，并返回最终金额；失败时不得创建订单；Creem 已知价格不调用金额核算接口。
- 视图层：源码不再包含旧核算文案和按钮；存在单一“立即充值”主操作及加载文案。
- 布局层：默认宽度双栏、最小宽度单列保持不变；统一卡片高度常量存在，两个卡片根容器都接受可扩展高度。
- 回归：二维码仍在软件内展示，不使用浏览器或定时轮询。

## 范围边界

- 不修改 new 后端支付接口。
- 不修改订阅购买弹窗的支付流程。
- 不新增支付状态轮询。
- 不重做资金记录与套餐卡片，避免把本次充值交互改造扩大成整页重构。
